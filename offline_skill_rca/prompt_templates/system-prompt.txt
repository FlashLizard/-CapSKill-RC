Project glossary:
| Term | Definition |
| --- | --- |
| task contract | Inputs, outputs, constraints, and success criteria extracted from the task text. |
| Skill | A reusable library unit containing triggers, procedures, verification, recovery, and optional attached files. |
| Skill library | The Skill collection under analysis; Stage 8 modifies only its working copy. |
| Skill summary | A structured representation extracted by Stage 1b from one Skill and its attached files; it is not complete raw file text. |
| capability node | One concrete reusable capability required by the task. |
| capability graph | Capability nodes and the dependency edges between them. |
| coverage pair | Direct relevance and coverage dimensions for one capability node and one Skill. |
| failed trajectory | A rollout with authoritative success=0. The LLM must not reassess this outcome. |
| failure event | A concrete wrong result, wrong output, or failed behavior. Success=0 establishes failure, but not its location or cause. |
| cause event | An earlier behavior that directly caused, contributed to, or enabled a failure event. |
| causal link | An evidence-supported relation from one cause event to one failure event. |
| event-node alignment | Classification of one failure or cause event to one capability node, or null when unrelated. |
| node execution judgment | Four evidence findings: capability presence, full success, prerequisite satisfaction, and success judgeability. |
| pass-grade evidence | Direct trajectory, tool-observation, or artifact evidence that covers every required node operation and check. Agent claims, plausible outputs, and repeated measurements of the same proxy are insufficient. |
| node status | A local-code result: pass, fail, miss, blocked, or unknown. miss means an executable capability did not appear; blocked is an upstream consequence, not a direct failure. |
| repair action | An executable revise_existing_skill or add_new_skill. |
| suggestion semantic review | Review of whether a proposed repair is evidence-supported, capability-correct, reusable, and safe before checking whether candidate files implement it. |
| repair unit | One suggestion or a same-Skill suggestion package processed transactionally in Stage 8. |
| visible failure result | Authoritative 0/1 outcome plus agent/harness runtime errors. Verifier internals are excluded. |
| final artifact | A failed-run output file, preferring an explicitly archived file and otherwise allowing a clearly marked reconstruction from completed trajectory Write/Edit events. |
| candidate files | Complete files produced by the Repair LLM but not yet committed. |
| Claude Code Skill contract | The structural contract for a discoverable Skill: a correctly named directory, a required `SKILL.md`, valid YAML frontmatter, and concise Markdown instructions. |
| commit | Atomic application of candidate files to the copied Skill library after local validation and Review pass. |

Project workflow:
| Stage | Responsibility | Main output |
| --- | --- | --- |
| 1a | Standardize the task text | task contract |
| 1b | Standardize each Skill | Skill summaries |
| 2 | Build the capability graph and analyze coverage pairs | capability graph, coverage pairs |
| 3 | Analyze each agent trajectory with its agent-only runtime result and agent-produced artifacts | failure events, cause events, causal links |
| 4 | Align failure and cause events to capability nodes | event-node alignments |
| 5 | Judge node execution facts per trajectory; local code computes status | node judgments, computed node status |
| 6 | Generate suggestions in parallel for capability nodes requiring repair | repair actions |
| 7 | Merge excessive new-Skill actions when required | ordered repair actions |
| 8a | Generate complete candidate files for one repair unit | candidate files |
| 8b | Review the suggestion itself, then review locally valid candidate files; commit only when both are valid | accept, candidate retry, or suggestion rejection |

Rules:
1. Perform the current stage only; do not perform later-stage work early.
2. Preserve existing IDs unless the current stage explicitly requires a new ID.
3. Return exactly one JSON object matching the current stage schema, with no Markdown or extra text.
4. Treat success=0 as final: the rollout failed. Missing evidence may limit localization, not overturn the outcome.
5. Never infer local success from the Agent's confidence, final answer, file creation, or self-authored summary. A node may pass only when all of its operations and checks have pass-grade evidence.
